## 概要

本プロジェクトは、Discord コミュニティの活動データを収集・蓄積・可視化するプラットフォームを構築する。

Discord API と Databricks Lakehouse を連携させ、活動データの継続的な取得・集計・可視化を実現する。  
**Phase 1** では活動傾向を「見える化」する。**Phase 2** では **LLM（Few-shot プロンプティング）の導入**と**ダッシュボードの Web サイト化・改良**、および **Bot の安定化**により、告知文生成や運営の意思決定支援を強化する。  
スコープは **Phase 1 完了後、2026年4月第3週までに Phase 2 まで完了**させる形で設定している。

本リポジトリは**データ基盤・可視化・Phase 2 の LLM とダッシュボード改良**に焦点を当てる。進捗の正は **Azure DevOps**（[docs/azure_devops_backlog_snapshot.md](docs/azure_devops_backlog_snapshot.md) でスナップショットを参照可能）。

## 本プロジェクトで作るもの

### コミュニティ施策のサイクル（4段階）

本プロジェクトでは、コミュニティ施策を次の **4 段階** に分解して課題整理と To-Be 設計を行っている。

1. **現状把握**（Current State Grasp）— コミュニティの状態を客観的に観測し、変化を検知する  
2. **設計**（Design）— データと仮説に基づき、優先順位と施策設計を行う  
3. **実行**（Execution）— 設計された施策を再現性をもって実行する  
4. **検証・改善**（Validate & Improve）— 施策の実行結果を定量的に検証し、得られた学習を次の設計に反映させる  

課題には前段階との依存関係があるため、**現状把握 → 設計 → 実行 → 検証・改善** の順で段階的に解決する。

## プロジェクト要件（要件定義）

### 1. プロジェクト概要

#### 1.1 プロジェクト名
Discord コミュニティ活動可視化プラットフォーム

#### 1.2 背景と課題
本プロジェクトの発起人は、600名超のメンバーを擁する大学 e スポーツ系 Discord コミュニティを運営している。  
定期的なイベントの企画やコミュニティの活性維持が重要である一方、参加率を最大化する日時の選定など、重要な意思決定は現状、経験や直感に大きく依存している。

実際には、「週末の方が参加者が多い」といった一般的な前提と異なる傾向も観察されている。  
しかし、そのような観察を定量的なデータとして体系的に記録・検証・共有する仕組みは存在しない。

#### 1.3 プロジェクト目的
本プロジェクトの目的は、Discord サーバーからユーザー活動データを収集・蓄積・可視化し、コミュニティ運営の意思決定を経験や直感ではなくデータに基づいて行えるようにすることである。

#### 1.4 JEDAI における位置づけ
本プロジェクトは、ビジネス以外のデータソース（具体的には Discord の活動ログ）を対象に、Databricks を用いたデータプラットフォームおよび可視化パイプラインを構築する実践事例として、JEDAI で紹介する。

#### 1.5 前提条件（Assumptions）

- Discord Developer Portal で Bot が作成でき、必要な Intent が有効にできる
- Databricks ワークスペースが利用可能で、Unity Catalog が有効である
- コミュニティ規模は現状のオーダー（数百〜数千メンバー程度）を想定する
- 運営メンバーが Bot Token の管理と取得手順を理解・実施できる

### 2. As-Is / To-Be

本プロジェクトでは、**施策サイクル**（現状把握 → 設計 → 実行 → 検証・改善）の4 つの Phase に沿い、各 Phase の As-Is 課題を整理した。  
全ての課題には**前段階の課題との依存関係**があり、順序立てて一つずつ解決する必要がある。

---

#### 2.1 As-Is（現状）

施策サイクルの4 つの Phase に沿い、各 Phase ごとに**現在のワークフロー**と**課題のマトリックス**を記載する。

---

##### 2.1.1 Phase 1：現状把握（Current State Grasp）— Dashboard 関連

**現在のワークフロー**

- コミュニティの状態は、**運営メンバーがサーバー内で活動するなかで得る「生の声」**に大きく依存している
- それ以外は**定性的な推測**がほとんどで、メモやスプレッドシートで数値や傾向を記録・集計することはほとんど行っていない
- 参加人数・メッセージ数などを Excel やスプレッドシートで継続的に記録することは**行っていない**。運営は有志であり業務ではないため、数値の継続的な記録まで手を回せていない

**課題のマトリックス**

| 課題 | 内容 |
|------|------|
| 全体アクティビティの俯瞰 | コミュニティ全体の活動状況を時系列で総合的に把握できていない。データの可視化・集計が不十分である。 |
| 活性度の構造的把握 | コミュニティの活性度がどのように変化しているかを、指標や構造を用いて分析・理解できていない。変化の背景や要因を捉えられていない。 |
| 定量指標のオペレーション化 | 定量指標の収集・管理プロセスが確立しておらず、体系的なデータ蓄積が行われていない。過去データとの比較や傾向分析が困難である。 |

---

##### 2.1.2 Phase 2：設計（Design）— ML 関連

**現在のワークフロー**

- 施策は、**大きな目的**（例：新歓でのメンバー定着、週1回の定期イベント）を達成するためにイベントを企画する形で進めることが多い
- 企画内容は、運営内で共有している**フォーマット**に担当者が記入する。フォーマットの項目は、イベント名（ゲームタイトルが分かるもの）、開催目的、日時、形式（個人参加 or チーム参加）、優勝賞金・ロール、詳細・コメント等である
- イベントの日時・内容は**運営メンバー複数名で相談**して決めている
- 学事など重要な行事と重ならないようにはしているが、**定量データはほとんど参照しておらず**、休日や「人が集まりやすそうな日程」を経験則で選ぶことが多い
- フォーマットが埋まり次第、**企画を固める → 日時を決める → 告知を作成する → フォームなど必要なものを用意する**という流れで進める

**課題のマトリックス**

| 課題 | 内容 |
|------|------|
| 施策の効果的な実施日の不明 | どのタイミングで施策を実行すれば最も効果的であるかの予測・判断ができていない。適切な実行タイミングを見極める分析やモデルが存在しない。 |
| 経験則依存のイベント設計 | イベント・施策の計画がデータや検証可能な仮説に基づいておらず、個人の経験や勘に頼っている。施策の成功が属人化し、再現性・最適化が難しい。 |
| 短期成果偏重 | 目先の成果や短期的な目標に意識が向き、コミュニティメンバーとの長期的な関係性を築く施策・戦略が不足している。持続可能なコミュニティ成長が阻害されやすい。 |

---

##### 2.1.3 Phase 3：実行（Execution）— LLM・RAG 関連

**現在のワークフロー**

- 告知は**アナウンス用チャンネル**に投稿し、Markdown を用いて参加しやすそうな雰囲気を意識している。必要に応じてボイスチャットでの勧誘や、人数を確保したいときの DM 送付も行っている
- 告知文の作成に**定型的なオペレーションはない**が、ChatGPT に過去の告知を入力するなどして、文面の品質をある程度そろえるようにはしている
- 投稿タイミングは、ゲーマーがアクティブになりやすい夜、学生がスマートフォンを確認しやすい昼休み・通学時間の朝などを**感覚で選んでいる**。ただし、インプレッションや反応などの**定量データは取得しておらず**、タイミングの効果は検証できていない

**課題のマトリックス**

| 課題 | 内容 |
|------|------|
| 告知・クリエイティブのゼロベース制作 | 告知文やデザイン素材の作成が、過去の資産やテンプレートを活用せず毎回最初から行われている。作業効率の低下と品質のばらつきが生じやすい。 |
| 告知タイミングの最適化ロジックの欠如 | 告知の配信タイミングを最適化するルールやアルゴリズムが存在しない。誰に・いつ・何を送るかの判断が属人的で、効果の最大化が図れていない。 |
| 告知の届く範囲の不足 | 施策や情報を受け取るべき対象者に、告知が十分届いていない。ターゲティングの精度が低く、情報伝達の効率が悪い。 |

---

##### 2.1.4 Phase 4：検証・改善（Validate & Improve）

施策の実行結果を定量的に検証し、得られた学習を次の設計に反映させる段階。単なる「評価」ではなく、検証結果に基づく改善アクションと設計へのフィードバックまでを含む。

**現在のワークフロー**

- イベント後は**振り返りを行うようにしている**。新メンバーには OJT の一環として上級生がメンターとなり、振り返りを実施する
- 記録は**定量データではなく**、今回の感想や本番での立ち回り・反省が中心で、ドキュメントに残している

**課題のマトリックス**

| 課題 | 内容 |
|------|------|
| KPI の非継続的モニタリング | 施策の成果を測る KPI が定期的・継続的に追跡・監視されていない。施策効果の定量的検証や改善につなげる機会を逸している。 |
| 学習内容の未蓄積・未共有 | 施策の実行や検証から得られた知見・教訓が、組織内で体系的に蓄積・共有されていない。過去の成功・失敗から学べず、ナレッジとして活用されていない。 |
| データに基づかない振り返り | 施策後の反省や効果検証が、具体的なデータや客観的事実に基づかず、主観や印象に偏っている。根本的な改善策の立案が困難である。 |

---

**As-Is の結論**  
上記の課題は相互に依存しており、現状把握 → 設計 → 実行 → 検証・改善の順で、一つずつ解決していくアプローチが有効である。

#### 2.2 To-Be（目指す姿の分解と Phase 定義）

---

##### 2.2.1 目指す4つの状態と Phase の分割

To-Be では、コミュニティリーダーが次の4つの状態を実現することを目指す。

1. **現状把握**：コミュニティの状態を客観的に観測し、変化を検知できる。
2. **設計**：データと仮説に基づき、優先順位と施策設計を行う。
3. **実行**：設計された施策を再現性をもって実行する。
4. **検証・改善**：施策の実行結果を定量的に検証し、得られた学習を次の設計に反映させる。

これらを **Phase 1（今回のスコープ）**、**Phase 2**、**Phase 3** に分け、各 Phase の達成イメージ・必要作業・Gap 分析を以下に示す。  
Gap 分析では、**As-Is の課題** → **解消に必要な取り組み**（抽象的なアイデア）→ **ソリューション**（Databricks のウィジェット・ML 手法等の具体的な取り組み）の3列で整理している。

---

##### 2.2.2 Phase 1：現状把握の基盤整備（今回のスコープ）

**達成時のイメージ（To-Be）**

- コミュニティ全体のアクティビティを時系列でダッシュボード上に俯瞰できる。
- 曜日・時間帯ごとの活動傾向や、活性度の変化を構造的（指標・集計）に把握できる。
- 定量指標の収集・蓄積がオペレーション化され、継続的にデータが蓄積される。
- データに基づく仮説形成や、直感とデータの乖離の確認が可能になる。

**必要な作業**

- Discord API による活動データの取得と、Databricks への継続的な蓄積。
- 取得スクリプトの実行場所と Databricks へのデータ受け渡し方法の設計。
- 活動傾向（曜日・時間帯等）の集計ロジックの実装。
- ダッシュボードによる可視化（全体アクティビティ・時系列・傾向）。
- 再現手順・設計意図の README 記載。

**Gap 分析（As-Is → Phase 1 To-Be）**

| As-Is の課題 | 解消に必要な取り組み | ソリューション（具体的な取り組み） |
|--------------|------------------------|--------------------------------------|
| 全体アクティビティを時系列で俯瞰できていない | ダッシュボードで時系列・集計を可視化する。 | Lakehouse ダッシュボードの **Time series** ウィジェット（日付軸×活動量）、**Line chart** でトレンド表示。 |
| 活性度の変化を構造的に把握できていない | 曜日・時間帯等の指標で集計し、傾向を可視化する。 | **Bar chart**（曜日別）、**Heatmap**（曜日×時間帯）または **Pivot table** で集計・傾向を可視化。 |
| 定量指標の管理がオペレーション化されていない | 取得〜蓄積〜集計のパイプラインを構築し、継続蓄積を可能にする。 | **Databricks Workflows** で取得ジョブをスケジュール、**Delta テーブル** に蓄積、**SQL 集計** 結果をダッシュボードに連携。 |

**Phase 1 完了後の業務フロー（改善後のオペレーション）**

- 運営メンバーは、イベント企画前や定期確認のタイミングで、Lakehouse ダッシュボードを参照する。
- ダッシュボードの Time series・Bar chart・Heatmap により、曜日・時間帯ごとの活動量やトレンドを把握する。
- イベントの日時候補を検討する際、経験則だけでなくダッシュボード上の傾向を根拠として参照する。
- 振り返り時は、施策前後の活動量や曜日別の推移をダッシュボードで確認し、定性的な印象を補足する。
- データは Workflows により日次で取得・蓄積され、手動での継続記録は不要である。

---

##### 2.2.3 Phase 2：LLM 導入とダッシュボード改良（今回のスコープ — 2026年4月第3週までに完了）

**達成時のイメージ（To-Be）**

- 告知文・クリエイティブのテンプレート化や、**Few-shot プロンプティング**を活用した LLM による半自動制作が可能になる（Phase 2 では RAG は用いない）。
- ダッシュボードが Web サイトとして利用可能になり、新たな特徴量・メトリクスで運営の判断材料が拡充される。
- Bot が安定稼働し、必要に応じて Discord に直接ポストできるようになる。

**必要な作業**

- 告知文・文面の資産化（**Few-shot プロンプティング**による LLM 活用。RAG は Phase 2 では使わない）。
- ダッシュボードの Web サイト化（新特徴量・新 Widget の追加、Databricks Apps へのデプロイ）。
- Bot の安定化（バグ修正、Docker 化、EC2 Linux 化、Discord 直接ポスト機能の追加）。

**Gap 分析（Phase 1 完了後 → Phase 2 To-Be）**

| As-Is の課題 | 解消に必要な取り組み | ソリューション（具体的な取り組み） |
|--------------|------------------------|--------------------------------------|
| 告知・クリエイティブが毎回ゼロベース | テンプレート・Few-shot による文面支援の検討。 | **Few-shot プロンプティング**（Databricks Foundation Model API 等）で文面・告知案を生成。RAG は用いず、プロンプトとサンプルで制御。 |
| ダッシュボードが Lakehouse 内にとどまり共有しづらい | ダッシュボードの Web 公開・デプロイの検討。 | **Databricks Apps** にダッシュボードをデプロイし、Web サイトとして利用可能に。新メトリクス・新 Widget を追加。 |
| Bot が不安定で Discord への直接投稿ができない | Bot の安定化と Discord 連携の強化。 | **Bot のバグ修正**、**Docker** 化、**EC2 を Linux** に変更。**Discord に直接ポストする機能**を追加。 |

**Phase 2 完了後の業務フロー（改善後のオペレーション）**

- 告知文・クリエイティブの作成は、**Few-shot プロンプティング**により LLM で文面案を生成する。運営メンバーが生成結果を確認・編集し、最終版を確定する（RAG は使わない）。
- ダッシュボードは **Databricks Apps** 上で Web サイトとして参照でき、新たな特徴量・メトリクスで活動傾向や施策効果を把握する。
- Bot は安定して稼働し、必要に応じて **Discord に直接告知をポスト**できる。
- 上記により、告知制作の負荷軽減と、ダッシュボード・Bot の利用しやすさが向上する。

##### 2.2.4 Phase 3：実行の再現性と最適化（今後解決すること）

**達成時のイメージ（To-Be）**

**必要な作業**

- 告知配信タイミングのルール化または最適化ロジックの実装。
- 対象者セグメントの定義と、配信対象の最適化。

**Gap 分析（Phase 2 完了後 → Phase 3 To-Be）**

| As-Is の課題 | 解消に必要な取り組み | ソリューション（具体的な取り組み） |

| 告知タイミングの最適化ロジックがない | Phase 2 の予測・設計と連携した告知スケジュール・配信ロジックの設計。 | **Databricks Workflows** で Phase 2 の予測結果を入力にしたスケジュール実行。**Alert** で条件に応じたトリガー設定。 |
| 告知が適切な対象に十分届いていない | セグメント定義と配信対象・チャネルの最適化。 | **セグメント定義テーブル**（Delta）と **SQL** で配信対象リストを生成。外部連携または簡易 API で対象者に届ける。 |

**Phase 3 完了後の業務フロー（改善後のオペレーション）**

- 告知文・クリエイティブの作成は、テンプレートを起点とし、必要に応じて RAG や Foundation Model API で文面案・素材案を生成する。運営メンバーが生成結果を確認・編集し、最終版を確定する。
- 告知タイミングは、Phase 2 の予測結果を Workflows や Alert で受け取り、ルールまたはスケジュールに従って決定する。条件に応じたトリガーで自動実行することも可能である。
- 配信対象は、セグメント定義テーブルと SQL で対象者リストを生成し、運営メンバーがレビュー・承認したうえで配信する。
- 上記の一連の流れは、Databricks Workflows によるスケジュール実行により、再現性をもって運用される。

##### 2.2.5 検証・改善の To-Be（全 Phase で段階的に強化）

**達成時のイメージ（To-Be）**

- KPI が継続的にモニタリングされ、施策の実行結果を定量的に検証できる。
- 学習内容が蓄積・共有され、次の設計に反映される。
- 振り返りがデータに基づいて行われ、改善策の立案につながる。
- **RAG により施策結果を自動的に検証**し、ナレッジベースと連携した評価が可能になる。
- **Action Item Recommendation** のような機能により、検証結果から次に取るべきアクションがレコメンドされる。

**必要な作業（Phase 1 から開始し、Phase 2・3 で拡張）**

- **Phase 1**：活動指標の可視化により、KPI モニタリングの基盤を整える。
- **Phase 2・3**：施策別 KPI の定義、振り返りプロセスの標準化、ナレッジベースや振り返りテンプレートの整備。RAG による自動検証、Action Item Recommendation 機能の検討・実装。

**Gap 分析（検証・改善まわり）**

| As-Is の課題 | 解消に必要な取り組み | ソリューション（具体的な取り組み） |
|--------------|------------------------|--------------------------------------|
| KPI が継続的にモニタリングされていない | ダッシュボードによる KPI 可視化と定期確認プロセスの確立 | Lakehouse ダッシュボードの **Counter** / **Time series** ウィジェットで KPI を表示、**Alert** で閾値監視 |
| 学習内容が蓄積・共有されていない | 振り返り成果の記録場所・形式の整備と共有ルールの策定 | **Notebook** または **Repos 内 Markdown** で振り返りテンプレート、**Delta テーブル** に振り返りメタデータ・リンクを保存。**RAG** で過去の学習を検索・活用 |
| 振り返りがデータに基づいていない | 振り返り時に参照する指標・データの明文化とデータ参照の習慣化 | ダッシュボードの **Filter** で施策別表示、**SQL クエリ** で事前に指標を算出し振り返り時に参照する手順の明文化。**RAG** による自動検証、**Action Item Recommendation** で次アクションをレコメンド |

**検証・改善の業務フロー（改善後のオペレーション）**

- 運営メンバーは、スプリント終了時やイベント後など定期的なタイミングで、ダッシュボードの KPI（Counter / Time series）を確認する。Alert で閾値監視が設定されていれば、異常時に通知を受ける。
- 振り返りは、Notebook または Markdown の振り返りテンプレートに従って実施する。ダッシュボードの Filter で施策別表示を切り替え、SQL クエリで事前算出した指標を参照しながら、データに基づいて効果検証と改善策を検討する。
- 学習内容（施策の結果、得られた知見、次回への教訓）は、振り返りテンプレートで記録し、Delta テーブルにメタデータ・リンクを保存する。運営内で共有し、次の設計に反映する。
- RAG による自動検証が整っている場合は、施策結果をナレッジベースと照合して評価する。Action Item Recommendation がある場合は、レコメンドされた次アクションを振り返りで検討し、採用・却下を判断する。

**予測ターゲットの方針（いずれかを選択）**

- **方針 A（活動量プロキシ）**: 曜日・時間帯の活動量（Phase 1 の `activity_by_weekday_hour` 等）を「施策が効きやすい日時」のプロキシとし、Phase 1 Gold のみで予測する。イベント別参加データは使わない。予測出力は「推奨曜日・時間帯」または活動量スコアとする。
- **方針 B（イベント参加データ）**: Phase 2 でイベント開催日時と参加数・参加率の対応データを取得・蓄積し、教師データとして利用する。取得元（Discord のイベント機能／手動登録／フォーム連携等）、Bronze/Silver/Gold の配置、教師ラベルの定義は設計時に決定する。

**長期指標の定義（設計時に確定）**

- **週次アクティブ（WAU）**: 直近 7 日間に 1 回以上活動したユニークメンバー数。
- **月次アクティブ（MAU）**: 直近 30 日間に 1 回以上活動したユニークメンバー数。

上記を採用した場合、Delta 集計と Dashboard の Time series でトレンド表示する（REQ-002-003）。

### 10. コミュニティリーダー能力の定義

本プロジェクトでは、*コミュニティリーダー*の役割を単一の責任として扱わない。  
代わりに、**コミュニティ施策のサイクル**（本要件定義の 2. As-Is / To-Be で用いた4 つの Phase）に沿ったコア能力に分解する。

これらの能力は、コミュニティを持続的に運営・成長させるために必要な基本機能を表し、As-Is 課題の整理および To-Be の Phase 定義と同一の枠組みで定義している。

#### 10.1 コミュニティリーダー能力モデル

コミュニティリーダーの役割は、次の**4つの能力**に分解する。

1. **現状把握**（Current State Grasp）
2. **設計**（Design）
3. **実行**（Execution）
4. **検証・改善**（Validate & Improve）

この分解は、相互に排他的で網羅的（MECE）になるよう設計しており、セクション2の As-Is / To-Be における4 つの Phase と対応している。Phase 4 は「評価」ではなく「検証・改善」とし、定量的な検証と改善アクション・次設計への反映までを含むニュアンスで扱う。

---

#### 10.2 能力の定義

##### 1. 現状把握（Current State Grasp）
コミュニティの状態を客観的に観測し、変化を検知する。

例：
- 全体アクティビティを時系列で俯瞰する
- 曜日・時間帯ごとの活動傾向や活性度の変化を構造的に把握する
- 定量指標を継続的に蓄積・管理する

##### 2. 設計（Design）
データと仮説に基づき、優先順位と施策設計を行う。

例：
- 施策を効果的に行える日時・ターゲットの検討
- データ・仮説に基づくイベント・施策の設計
- 短期成果と長期の関係性構築のバランスを考慮した優先付け

##### 3. 実行（Execution）
設計された施策を再現性をもって実行する。

例：
- 告知文・クリエイティブの制作と配信
- 告知タイミングの最適化と適切な対象への届け方の実施
- イベント・コミュニティ活動の調整

##### 4. 検証・改善（Validate & Improve）
施策の実行結果を定量的に検証し、得られた学習を次の設計に反映させる。

例：
- KPI の継続的なモニタリングと施策効果の定量検証
- 振り返りの実施と学習内容の蓄積・共有（次設計へのインプット）
- データに基づく振り返りと改善策の立案
- To-Be：RAG による自動検証、Action Item Recommendation による次アクションのレコメンド

## 開発・標準

本プロジェクトは **Databricks Solution Standard Guideline** および `.cursor/rules/` のプロジェクトルールに従う。

- **データ変換**はメダリオン構成（Bronze → Silver → Gold）に従う
- **スクリプト・ノートブック**には説明コメントを含める（[コメント標準](standards/01-language/CODE_COMMENTING_STANDARD.md)）
- **シークレット**（トークン、キー）はソースコード外で管理（Databricks Secrets）
- Phase 1 では**再現性**を性能最適化より優先する

### 主な標準

- **命名**：[myteam 命名規則](standards/00-core/myteam_Naming_Conventions.md)
- **コメント**：[コメント標準](standards/01-language/CODE_COMMENTING_STANDARD.md)
- **エラー**：[エラーハンドリング](standards/03-patterns/ERROR_HANDLING_STANDARD.md)
- **パターン**：[実装パターン](standards/03-patterns/IMPLEMENTATION_PATTERNS.md)

### アーキテクチャ

- **Bronze**：追加のみ、生データを保持（例：Discord API レスポンス）
- **Silver**：クレンジング、検証、必要に応じて SCD Type 2、DLT を推奨
- **Gold**：集計・特徴量テーブル、必要に応じて `overwriteSchema` で上書き
- **データ品質**：DLT Expectations、不正データは命名規則に従い検疫テーブルへ

## 設計判断

本プロジェクトにおける主な設計判断は以下のとおりである。詳細は [プロジェクト要件（要件定義）](#プロジェクト要件要件定義) および各標準を参照。

- **メダリオン構成**：Bronze（生データ）→ Silver（クレンジング・検証）→ Gold（集計・分析）の 3 層構成とする。スキーマ進化・再計算性を考慮する。
- **Silver 層で DLT を推奨**：Delta Live Tables によりクレンジング・検証・Expectations を宣言的に定義し、検疫テーブルへ不正データを分離する。
- **Gold 層で overwriteSchema**：集計テーブルはフルリフレッシュを想定し、`mode("overwrite")` と `option("overwriteSchema", "true")` でスキーマ衝突を避ける。
- **シークレットを Databricks Secrets で管理**：Discord Bot Token 等の認証情報は Databricks Secrets に保存し、ソースコードやリポジトリにコミットしない。セキュリティと再現性を両立する。

## 開発環境セットアップ

### 1. リポジトリのクローン

```bash
git clone <リポジトリURL>
cd jedai_pj
```

### 2. Databricks の設定

```bash
pip install databricks-cli
databricks configure --token
```

### 3. 環境の準備

```bash
python -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate
pip install -r requirements.txt   # 存在する場合
```

### 4. 環境変数・シークレット

- `DATABRICKS_HOST`、`DATABRICKS_TOKEN`、カタログ・スキーマを必要に応じて設定
- Discord API の認証情報は Databricks Secrets に保存し、リポジトリにコミットしない
- **Discord 活動収集ボット**（`scripts/01_setup/bot.py`）を動かす場合は、**リポジトリルート**から起動すること。例: `cd jedai_pj && python scripts/01_setup/bot.py`。出力は `data/bronze/` に保存される。環境変数 `DISCORD_BOT_TOKEN` を設定すること。

### 5. セットアップスクリプトの実行

以下の順で実行する。

```bash
scripts/01_setup/    # カタログ、スキーマ、ロール、メタデータテーブル
scripts/02_bronze/   # Bronze 取り込み（Discord → Delta）
scripts/03_silver/   # Silver クレンジング（DLT またはレガシー）
scripts/04_gold/     # Gold 集計
scripts/06_dashboards/ # ビュー・ダッシュボードクエリ
scripts/07_workflows/  # Databricks Workflows 定義・実行（手動実行またはスケジュール実行でパイプライン実行）
```

### 6. 動作確認

- Databricks ワークスペースおよび Unity Catalog へのアクセス
- 少なくとも 1 回のパイプライン実行（Bronze → Silver → Gold）
- ダッシュボードが期待どおりに表示されること

## トラブルシューティング

### Databricks 接続

- `DATABRICKS_HOST` と `DATABRICKS_TOKEN`（またはプロファイル）を確認
- トークンの有効期限を確認し、再認証後にリトライ
- Unity Catalog およびワークスペースの権限を確認

### DLT / パイプラインエラー

- テーブル名の衝突がないか確認、開発時は必要に応じて `_dlt` を使用
- [DLT 完全ガイド](guides/implementation/DLT_COMPLETE_GUIDE.md) と Expectations を確認
- ドロップされた行は検疫テーブルで確認

### スキーマ・書き込みモード

- Bronze/Silver：`mode("append")` と `option("mergeSchema", "true")`
- Gold：フルリフレッシュを想定する場合は `mode("overwrite")` と `option("overwriteSchema", "true")`
- 本番パイプラインでは DROP TABLE + CREATE TABLE を避ける

### Discord API

- トークンと権限（例：Guild Members Intent）を確認
- レート制限とバックオフを考慮、認証情報は Databricks Secrets に保存
